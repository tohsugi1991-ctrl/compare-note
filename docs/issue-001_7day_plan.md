# ISSUE-001 7日間実装計画

前提: [issue-001_mvp_spec.md](./issue-001_mvp_spec.md)・[issue-001_user_flow.md](./issue-001_user_flow.md)・[issue-001_data_schema.md](./issue-001_data_schema.md)の設計に基づく。この計画書自体はまだ実装に着手しておらず、次に読む人がDay1からそのまま着手できることを目的とする。

---

## Day 1: セットアップ + データ層

- **作業内容**: 新規Viteプロジェクトを作成する。`IdeaForge/founder-os`のtsconfig・tailwind設定・oxlint設定を雛形として流用(ロジックはコピーしない)。`src/types.ts`にProject/CandidateResponse/Decisionの型を定義。`src/lib/storage.ts`にlocalStorageの読み書き関数(load/save/CRUD)を実装
- **成果物**: `npm run dev`で起動するVite+React+TSプロジェクトの雛形。`src/types.ts`、`src/lib/storage.ts`
- **完了条件**: ブラウザコンソールからダミーのProject/CandidateResponse/Decisionを書き込み、リロード後も読み込めることを確認できる
- **やらないこと**: UIデザイン、画面コンポーネントの実装、比較ロジック

## Day 2: 画面1(案件一覧)

- **作業内容**: 案件一覧画面(ProjectsScreen)を実装。案件作成フォーム(タイトル必須、context任意)、一覧表示(タイトル・更新日・決定状態バッジ)、空状態の表示
- **成果物**: 案件を作成 → 一覧に表示 → クリックで画面2へ遷移(画面2はまだ空でよい)までの動作
- **完了条件**: 案件を3件作成し、ブラウザをリロードしても一覧が復元されることを確認
- **やらないこと**: 検索・フィルタ・並べ替えUI、削除以外の編集機能

## Day 3: 画面2 状態A(候補追加・比較表示)

- **作業内容**: 案件詳細画面のうち「未決定」状態を実装。候補回答の追加フォーム(source選択+content貼り付け)、複数候補のレスポンシブな比較表示(ノートPC幅で横並び、狭い画面では縦積み)
- **成果物**: 1つの案件に2〜4件の候補を貼り付けて、画面上で読み比べられる状態
- **完了条件**: 候補を追加 → 表示 → リロードしても保持されることを確認。横並び表示が画面幅で崩れないことを確認
- **やらないこと**: 決定の記録機能、自動比較・自動要約、候補の並べ替えUI

## Day 4: 決定記録機能 + 状態Bへの切り替え

- **作業内容**: 候補の1つを「採用」として選ぶUI、decisionSummary(任意)とreason(必須)の入力、「決定を保存」ボタン。保存後は画面2を状態B(決定済み)へ切り替え、決定サマリーカードを表示。「編集する」から状態Aの決定セクションへ戻れるようにする
- **成果物**: 比較 → 採用選択 → 理由記入 → 保存 → サマリー表示、の一連が動作する状態
- **完了条件**: 決定済み案件をリロード・再訪しても決定内容が正しく表示される。決定後に編集して再保存できる(上書きされ、過去の値は残らない仕様通り)
- **やらないこと**: 決定履歴のタイムライン、複数決定の同時保持

## Day 5: 通し動作確認 + 自分自身での実案件投入(dogfooding)

- **作業内容**: 自分が実際に抱えている進行中の案件(白子町次期総合計画・香取市案件など)のうち最低2件で、実際にComparisonToDecisionフローを使う。UIの分かりにくい箇所・レスポンシブ崩れ・タイポを修正。localStorage容量超過時や極端に長いテキスト貼り付け時の最低限のエラー表示を追加
- **成果物**: 自分自身の実データで2件以上の決定記録が残っている状態。発見したバグの修正
- **完了条件**: 「これなら他人にも見せられる」と感じる品質に到達している。作成→比較→決定→再訪確認の一連を3回以上繰り返し、クラッシュ・データ消失が起きないことを確認
- **やらないこと**: 新機能の追加、装飾的なデザインの作り込み(動作の安定を優先する)

## Day 6: デプロイ + 告知文の準備

- **作業内容**: Vercelへデプロイし本番URLを確認。favicon等の最低限の体裁を整える。「データはこのブラウザだけに保存されます」という一言をUIに追加(データ層の制約をユーザーに正直に伝えるため)。[issue-001_launch_plan.md](./issue-001_launch_plan.md)で使う告知文の下書きを仕上げる
- **成果物**: 公開URL(本番)、告知文の下書き
- **完了条件**: 自分以外の1名(家族等でよい)に本番URLで実際に操作してもらい、迷わず「案件作成→比較→決定」まで完了できることを確認する簡易ユーザビリティチェックを実施済み
- **やらないこと**: 課金導線、追加機能の実装

## Day 7: 公開・初期ユーザーへの案内

- **作業内容**: 告知文をX/noteに投稿。知人・個人開発者コミュニティ・Claude Code利用者コミュニティへ直接案内し、10人への周知を完了する。フィードバック回収先(DMまたは簡易フォームのリンク)を明記して依頼する
- **成果物**: X/noteへの公開投稿、初期10人への個別案内の送付記録
- **完了条件**: 10人への案内が完了し、最初の反応(閲覧または利用開始)が最低1件確認できる
- **やらないこと**: 追加開発、バグ修正以外の機能変更
