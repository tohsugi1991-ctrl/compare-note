## 1. 誰のどの場面を解決するか

**対象ユーザー**(`JTBD_report.md` 1節・タスク指定条件と一致):

- ChatGPT・Claude・Gemini等を2種類以上使う
- AIを仕事・開発・調査・執筆のいずれかに週3回以上使う
- 複数AIの回答を比較することがある
- 後日、以前の判断や採用理由を探し直すことがある

具体的には、2〜6件程度の案件を並行して抱えるコンサルタント・企画職・個人開発者・フリーランスのライター/エンジニア。一般消費者全体や企業チームは対象外。

**解決する場面**(`JTBD_report.md` 4節「途中で発生する困りごと」より):

複数AIに同じ・類似の問いを投げて回答を比較したが、比較の記録を残さずに1つを採用してしまい、数日後に案件を再開したときに「なぜあの回答を採用したのか」「他に何を検討して捨てたのか」を思い出せず、比較からやり直す場面。JTBD_reportはこれを「比較」「再利用」という2段階のOpportunity Score(ともに16点、全段階中最高)として特定している。

## 2. なぜ単純なプロンプト保存アプリにしないのか

`complaint_comparison.csv`のISSUE-001初期MVP案(「案件タグ」ボタンでプロンプト+出力をワンクリック保存するChrome拡張)は、JTBD分析より**前**の時点の仮説であり、JTBD_report 15節で明示的に3案(案A:再開ブリーフィング/案B:比較して決める/案C:過去の型の呼び出し)に整理し直された結果、**案Bが最初に解くべきジョブ**と結論づけられている(理由: 案Bが生み出す意思決定データなしには案Cが成立しない、比較段階が最も満足度が低く重要度が高い、既存の大手無料サービスが構造的に解決できない領域である)。

本仕様書はこの案Bをそのまま実装する。「プロンプトを保存する」ことではなく、「比較して、何を採用し、なぜ決めたかを記録する」ことが主要価値になる。

### ISSUE-002(除外済み)との違い

`synced-opportunities.json`のISSUE-002(同じ質問を複数AIに投げて横並び比較)は、TypingMind等の既存プレイヤーが強く「現時点で除外」と判定済み。ISSUE-002は**AIへの自動問い合わせ・API統合による横並び比較**を指すが、本MVPは**すでに得た回答を人間が手動で貼り付け、比較・決定を記録する**という別のジョブ段階を扱う。API統合が不要なため、ISSUE-002を除外した理由(API統合のコモディティ化、TypingMindの先行)はそのまま当てはまらない。

## 3. MVPで検証する最大仮説

主仮説:

> 複数AIの回答を並べ、採用した回答と採用理由を短時間で記録できれば、ユーザーは後日の再開時に価値を感じ、繰り返し使う

3つに分解する:

| # | 仮説 | 何を検証するか |
|---|---|---|
| H1(行動仮説) | 複数の案件を抱える忙しいユーザーでも、回答を貼り付けて比較し、決定理由を記録する一手間を許容し、実際に記録行動を最後まで完了する | 記録という行為自体が新たな「転記の面倒くささ」を生まないか |
| H2(価値仮説) | 案件を再開したときに過去の決定記録を見返すと、「迷いなく再開できた」という体感的価値を感じる | JTBD_report 3節の成功定義(意思決定への自信)が実際に生まれるか |
| H3(継続仮説) | 上記の体感が、7日以内に同一ユーザーが2件以上の案件で繰り返し使う行動として表れる | 単発の物珍しさで終わらず、習慣化の兆しがあるか |

---

## 4. 実装する唯一の主要価値

**複数AIの回答を横並びで比較し、採用理由を短時間(1〜3行)で記録できること。**

保存でも検索でもタグ管理でもなく、「比較 → 採用 → 理由の記録」という一連の流れを1箇所に集約する一点のみに絞る。

---

## 5. MVPの主要機能(唯一の主要価値に必要な範囲のみ)

タスクで提示された5ステップのうち、主要価値の検証に不要なものは削らない(このMVPでは5つすべてが「比較して決める」というジョブの検証に必須)。

1. 案件を作る
2. 複数AIの回答を貼り付けて横並び比較する
3. 採用案を1つ選ぶ
4. 採用理由を1〜3行で残す
5. 後から案件を開き、決定内容を確認する

削った、または最小化したもの:

- 候補回答の並べ替え(ドラッグ&ドロップ)は実装しない。貼り付けた順に表示するのみ
- 決定の履歴(過去の決定を巻き戻す機能)は持たない。決定は1案件につき常に最新の1件のみ(上書き編集可能)
- 案件の検索・フィルタ・タグ・フォルダは持たない

## 6. 禁止機能(再掲)

以下はv1で一切実装しない: AI APIによる自動回答取得/ChatGPT・Claude・Geminiへのログイン連携/ブラウザ履歴の自動取得/Chrome拡張/チーム共有/コメント/通知/クラウド同期/高度な検索/タグ管理/フォルダ階層/課金/サブスク/SNS機能/Notion・Slack・Google Drive連携/自動要約/自動比較/自動推薦/モバイルアプリ。

ユーザーがAIの回答を手動で貼り付けるローカルWebアプリが基本形。

## 7. 画面数の結論:2画面

推奨構成は3画面(案件一覧/比較・意思決定/決定内容の確認)だったが、「比較・意思決定」と「決定内容の確認」は同一データ(Decision)の異なる状態(未決定/決定済み)にすぎず、別画面に分ける必然性がない。IdeaForge本体の設計判断(Screen 2を状態変化だけで4状態カバーする)と同じ考え方を踏襲し、**2画面**に絞る。詳細は[issue-001_user_flow.md](./issue-001_user_flow.md)。

1. 案件一覧
2. 案件詳細(比較・決定・確認を状態で切り替える1画面)

## 8. 技術構成の結論

### Founder OS流用 vs 別プロジェクト

比較した結果、**別プロジェクトとして新規に作る**。

| 観点 | Founder OS流用 | 別プロジェクト |
|---|---|---|
| ドメインの一致 | 低い。Founder OSは「自分の複数プロジェクトの優先順位判断」ダッシュボードで、Project/Opportunity型やsync-idea-forge-data.tsの同期ロジックと今回のCandidateResponse/Decisionは無関係 | 高い。ゼロから素直なデータモデルを組める |
| 公開の可否 | Founder OSは「新しいクラウドサービスは増やさず、ローカル完結で運用する」方針が明記済み(`founder-os/resume.md`)。この方針を破って公開用にVercel等へデプロイするのは既存方針との矛盾になる | 制約なく公開できる |
| 実装速度 | 既存コードを剥がす作業が発生し、むしろ遅くなる可能性 | 雛形(package.json/tsconfig/tailwind config/oxlint設定)だけ流用すれば速い |

**結論**: Founder OSには機能追加しない。`compare-note/`という新規ディレクトリ(このリポジトリ)を作り、Founder OSの設定ファイル一式(tsconfig・tailwind・oxlint等)だけを雛形として流用する。ビジネスロジック・型定義・コンポーネントはゼロから書く。

### スタック

- React + TypeScript + Vite + Tailwind CSS
- 状態: localStorageのみ(バックエンドなし/APIサーバーなし/認証なし/データベースなし)
- ルーティング: 使わない(URLルーティングライブラリなし)。画面切り替えはApp内のstateで行う(2画面のみのため十分)

詳細なデータ構造は[issue-001_data_schema.md](./issue-001_data_schema.md)を参照。

## 10. 最終判断(1ページサマリー)

- **作るもの**: 複数AIの回答を貼り付けて横並び比較し、採用した回答と理由(1〜3行)を記録し、後日案件を開くと決定内容がすぐ確認できるローカルWebアプリ「AI比較ノート」(仮称)
- **作らないもの**: プロンプト保存・検索・タグ・フォルダ・AI API連携・Chrome拡張・チーム共有・通知・クラウド同期・課金・自動比較/要約/推薦(6節に列挙したものすべて)
- **ターゲット**: AIを2種類以上・週3回以上使い、複数AIの回答を比較し、後日過去の判断を探し直すことがある個人(案件を2〜6件並行するコンサル/企画職/個人開発者/フリーランス)
- **最大仮説**: 比較と採用理由を短時間で記録できれば、後日の再開時に価値を感じ繰り返し使う(H1行動/H2価値/H3継続の3分解)
- **MVPの主要機能**: 案件作成 → 複数AI回答の貼り付け比較 → 採用案の選択 → 採用理由の記録 → 再訪時の決定内容確認、の5ステップのみ
- **画面数**: 2画面(案件一覧/案件詳細)
- **技術構成**: React+TypeScript+Vite+Tailwind CSS+localStorage、バックエンドなし。Founder OSには機能追加せず別プロジェクトとして新規作成(設定ファイルのみ雛形流用)
- **7日後の成功条件**: 公開済み・実利用者5人以上・案件作成10件以上・決定記録完了6件以上・24時間以上空けた再訪者3人以上・自由記述フィードバック5件以上・「今後も使いたい」3人以上(詳細は[issue-001_validation_plan.md](./issue-001_validation_plan.md))
- **Kill Criteria**: 貼り付け作業の離脱率60%以上/決定完了率40%未満/7日以内の再訪者が10人中2人未満/「Notion等で十分」という声が過半数/「比較より別の課題」という声が過半数、のいずれかがフィードバック5件以上到達時点で成立した場合はPivotまたは中止を検討
- **初期10人の集め方**: 知人3〜4人への直接連絡、X/noteでの開発ログ発信、個人開発者・Claude Code利用者コミュニティへの投稿(広告は使わない)。詳細は[issue-001_launch_plan.md](./issue-001_launch_plan.md)
- **Founder OSへ組み込むか別プロジェクトか**: 別プロジェクト(`compare-note/`)。理由は8節
- **次に実行すべき1件**: [issue-001_7day_plan.md](./issue-001_7day_plan.md) Day1「セットアップ+データ層」に着手する(新規Viteプロジェクトの作成とlocalStorage CRUD関数の実装)
