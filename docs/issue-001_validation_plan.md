# ISSUE-001 検証計画:成功条件とKill Criteria

7日後の成功は「公開できたか」ではなく検証結果で定義する。測定手段はアプリ内に分析コードを組み込まず(禁止機能である「クラウド同期」「バックエンド」を増やさないため)、初期10人への直接ヒアリング(DM・簡易フォーム)で行う。

---

## 測定方法(前提)

- localStorageのみで動くため、アプリ側からユーザー行動を自動計測する仕組みは持たない
- 初期10人には個別に案内するため([issue-001_launch_plan.md](./issue-001_launch_plan.md)参照)、Day3〜4とDay7〜10の2回、DMまたは短いGoogleフォームで直接ヒアリングする
- ヒアリング項目: 「アプリを開いたか」「案件をいくつ作ったか」「決定(採用理由)まで記録した案件はいくつか」「再度開いたか」「今後も使いたいか」「自由記述の感想」
- フィードバック母数が5件に満たない場合、Kill判定は保留し観察期間を1週間延長する(母数が少なすぎる状態で中止判断をしない)

---

## 成功条件(Day7時点)

| 項目 | 目標値 | 備考 |
|---|---|---|
| 公開できたか | Yes | Vercel等に本番URLがあり、第三者がアクセスできる状態 |
| 実ユーザー数 | 案内した10人のうち5人以上がアプリを開いて何らかの操作をした | 起動率50%が最低ライン |
| 比較案件の作成数 | 合計10件以上(全ユーザー合算) | 1人あたり平均2件が目安 |
| 決定記録の完了数 | 6件以上(作成された案件の60%以上が決定まで到達) | H1(記録の一手間を許容できるか)を直接測る指標 |
| 再訪または再利用の兆候 | 初回利用から24時間以上あけて再び開いた人が3人以上 | ヒアリングでの自己申告ベース(自動計測なしのため) |
| 自由記述フィードバック | 5件以上回収 | Kill Criteria判定の母数を満たすために必須 |
| 「今後も使いたい」と答えた人数 | 3人以上 | H2/H3(価値・継続仮説)の直接的な裏付け |

いずれも小さく現実的な目標であり、これを超えたら即座に大きく投資するという意味ではなく、「次の1週間も続ける価値があるか」を判断するための最低ラインとして設定している。

---

## Kill Criteria(中止・ピボット条件)

フィードバック5件以上が集まった時点で、以下のいずれかに該当すれば中止またはピボットを検討する。

| # | 条件 | 数値基準 | 該当した場合の対応 |
|---|---|---|---|
| 1 | 比較回答を貼り付ける作業を面倒だと感じる | 案件を作成したが候補回答を2件以上貼り付けないまま離脱した人が、ヒアリング対象5人中3人(60%)以上 | 貼り付け操作自体の摩擦が高すぎると判断。UIの簡略化で解決するか、案A(再開ブリーフィング)など別のジョブへのピボットを検討 |
| 2 | 決定理由を記録しない | 候補を2件以上貼り付けて比較した案件のうち、決定(reason記入)まで到達したのが40%未満 | 「比較はするが記録は面倒」という構造的な問題。H1棄却に相当し、記録の手間をさらに削れないか再検討、それでも改善しなければ中止 |
| 3 | 一度使った後に再訪しない | 初回利用者のうち、7日以内に別の日に2回目のセッションで開いた人が10人中2人未満 | H2/H3(再訪時の価値)が成立していない。JTBD自体の解釈が誤っている可能性があり、案Bの前提を再検証 |
| 4 | 既存のメモやNotionで十分と言われる | フィードバック5件中3件以上で「Notion/メモで十分」という趣旨のコメントが出る | 差別化が使用者に伝わっていない、または実際に差別化できていない。中止、または「案件の文脈保持」の価値をより明確に打ち出す方向へピボット |
| 5 | 比較よりも別の課題が強いと判明する | フィードバック5件中3件以上で「比較より○○が欲しい」という別ニーズの言及がある | 案Bより案A・案Cのほうが優先度が高い可能性。JTBD_reportの再評価が必要 |

**判定タイミング**: Day7を基本の判定日とするが、フィードバック母数が5件に届かない場合はDay14まで延長して再判定する。5件のいずれかの条件に単独で該当しただけで即中止はせず、複数条件(2つ以上)に該当した場合に中止、1つのみ該当した場合はピボット(UI改善・訴求の変更)を優先する。
