# Day7 Light Validation — Phase1: N=1 Dogfoodingレビュー

- 実施日: 2026-07-15
- 位置づけ: 当初のDay7計画(初期10人への案内)は今回実施しない。代わりに「N=1 dogfooding再検証 → 競合比較 → AIレビュー → 判定」の軽量検証(Light Validation)を行う。詳細な位置づけは[issue-001_7day_plan.md](./issue-001_7day_plan.md) Day7節を参照。
- 本レポートはDay5([day5_dogfooding_report.md](./day5_dogfooding_report.md))と同じ方法論・同じ注意書きを踏襲する。

## 0. 実施方法(先に断り書き、Day5と同様)

1. **実案件の中身は本物**。白子町P14の分割・再配置論点(`municipal_proposal_factory/projects/白子町/P14/human_decision_brief.md`)、香取市のPowerPoint作成Go/No-Go判定(`municipal_proposal_strategy_ai/projects/香取市/blueprint_completion_report.md`)、proposal_pattern_libraryの次セッション着手順位(`proposal_pattern_library/RESUME.md`)は、いずれも実在する進行中プロジェクトの実際の検討資料から要約・引用した。Day5で使った3案件(白子町総合計画・香取市KPI診断・IdeaForge Phase6)とは別の論点を選び、同じ案件の使い回しを避けた。
2. **候補回答はDay5と同様、実際の検討資料にある2つの現実的な方向性をそのまま2候補として登録した**(例: P14は`page_optimization_review.md`が挙げる4案のうち対照的な2案[案A/案D]を採用)。sourceラベルは`Claude Code`/`Claude`を使用し、異なるベンダーのAIを比較する本来の主要シナリオそのものの検証ではない点はDay5と同じ限界として明記する。
3. **入力操作はPlaywrightによる実ブラウザ自動操作(Chromium、永続プロファイル)で行った**。案件作成→候補2件登録→比較→採用選択→採用理由記録→一覧へ戻る、をすべて実ブラウザ上で実行し、スクリーンショットで見た目を確認した。ブラウザプロセスを一度終了し、同一の永続プロファイル(同一localStorage)で再起動して3案件を再訪し、決定内容が保持されていることを確認した。自動操作の実行時間(ミリ秒単位)は参考値として5節に残すが、本文の時間評価は文字数・操作数から見積もった**推定人間所要時間**を用いる。

### 案件1: 白子町P14 分割・再配置の採否

- **入力にかかった時間(推定)**: 約80秒(タイトル+context入力 約20秒、候補2件の貼り付け 約55秒 ※1件目のsourceが「Claude Code」でOther欄への追加入力が発生、候補content合計546文字とDay5よりやや長め)
- **比較にかかった時間(推定)**: 約45秒(2件・合計546文字を読み比べ)
- **決定記録にかかった時間(推定)**: 約75秒(採用選択+理由197文字+要約39文字。4案(A/B/C/D)から2案に絞った上でどちらを採るかという判断そのものに時間がかかった)
- **再訪時に決定内容を理解するまでの時間(推定)**: 約20秒
- **貼り付け作業の面倒さ**: 中。1件目(Claude Code)がOther経由になった
- **横並び比較のしやすさ**: 高。2カラムグリッドで両案のメリット・デメリットを視線移動少なく比較できた

- **decisionSummaryの有用性**: 高。「データ取得後に分割要否を再検討」という後で忘れがちな留保事項を残せた
- **既存のメモやNotionより良かった点**: 不採用案(案A: 分割+再配置)のメリットも画面に残るため、後日D-1〜D-4が取得できた際に「なぜあの時点では見送ったか」を再確認できる

## 3. 再訪確認の詳細

3案件の決定保存後、ブラウザプロセス(Playwright永続プロファイル)を一度終了し、再起動して同一localStorageを引き継いだ状態で一覧から3案件を開き直した。

- 3案件とも一覧の「決定済み」バッジ・タイトル・更新日時が正しく残っていた
- 各案件を開くと、決定カード(採用元source・content・採用理由・decisionSummary・決定日時)が画面最上部に即座に表示され、比較セクションまでスクロールしなくても「何を選び、なぜ選んだか」が分かった
- 実測(自動操作、参考値): 案件を開いてから決定カードが描画されるまで32〜36ms

この結果はDay5に続き、H2(価値仮説: 再開時に「迷いなく再開できた」という体感)を再現的に支持する。

## 7. Day5からの新規発見: exportData/importDataが実装済みだがUI未接続

Day5レポートにはなかった発見として、`src/lib/storage.ts`に`exportData()`/`importData()`がすでに実装され、`src/lib/storage.test.ts`でユニットテストも通過している(462〜491行目)にもかかわらず、`ProjectsScreen.tsx`/`ProjectDetailScreen.tsx`のどちらにもこれらを呼び出すUIボタンが一切存在しないことをコードレビューで確認した。README.mdは「エクスポート・バックアップ機能は現時点ではありません」と記載しているが、これは「UIから使えない」という意味では正しいものの、「データを取り出す手段が実装レベルで存在しない」という意味では正確ではない。localStorageのみでバックエンドを持たない設計上、この機能をUIに繋ぐコストは低い一方、繋がっていないことによるデータ喪失リスクは実在する。詳細と優先度は[ai_design_review.md](./ai_design_review.md)のMust Fixを参照。
