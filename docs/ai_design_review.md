# Day7 Light Validation — Phase3: AIレビュー(5人格)

[day7_light_validation.md](./day7_light_validation.md)(dogfooding結果)と[competitive_analysis.md](./competitive_analysis.md)(競合比較)、および実際のソースコード(`src/`)を根拠に、PM・UXデザイナー・個人開発者・SaaS創業者・投資家の5つの視点で批判的にレビューする。

---

## 1. PM

**良い点**: スコープが極めて絞られており(画面2つ、機能一覧が明確)、7日で公開まで到達した。実案件6件(Day5の3件+Day7の3件)でMust Fix該当のバグが継続してゼロ件という、仮説検証としては綺麗な結果が出ている。

**悪い点**: 「初期10人への案内」というDay7本来のマイルストーンを、今回また先送りしている(Day6完了→Day7 Light Validationへ切り替え)。dogfoodingは創業者自身の主観に強く依存した検証であり、他人による検証は今の時点で依然としてゼロ件のままである。

**最大リスク**: 実ユーザーの反応を一度も観測しないまま、「価値がある」という思い込みだけでv0.2の機能追加や告知文の調整に進んでしまうこと。N=1のdogfoodingがどれだけ整った結果を示していても、それは「作った本人が自分の道具を気に入っている」以上の証拠にはまだなっていない。

**最大価値**: 「採用理由を書かないと決定を保存できない」という設計上の強制力が、他ツールでは代替しにくい独自の価値になっている点。これは今回の6案件すべてで再現され、偶然ではないと言える水準にある。

**改善案**: Light Validationの後、必ず最低3人の第三者(知人・個人開発者コミュニティ)に実際に触ってもらうことを次の必達マイルストーンとして固定する。dogfoodingの結果が良かったことを理由に、外部検証を省略しない。

## 2. UXデザイナー

**良い点**: 決定済みカードが画面最上部に固定表示され、再訪時に「何を選び、なぜか」が一目で分かる情報設計は一貫している。比較カードの2カラムグリッド/横スクロール切り替えも画面幅に応じて適切に機能している。

**悪い点**: 「回答を追加」フォームが決定後も常に比較セクションより上に固定表示される(SF-1、Day5で発見済みかつ未修正)。決定済みの案件を後から見返すだけのユーザーにとって、この空フォームは明確なノイズになっている。sourceのプリセットに実際の主要な利用ソース(Claude Code)が無いのも、UIが実際の利用実態に追いついていない証拠であり、Day5(6候補中3回・50%)からDay7(6候補中6回・**100%**)へ悪化している。

**最大リスク**: `src/lib/storage.ts`には`exportData()`/`importData()`が実装され、`storage.test.ts`でユニットテスト(462〜491行目)まで通過しているにもかかわらず、`ProjectsScreen.tsx`/`ProjectDetailScreen.tsx`のどちらにもこれらを呼び出すUIが一切存在しない。README.mdは「エクスポート・バックアップ機能は現時点ではありません」と説明しているが、これはユーザー向けの説明としては現状正しい一方、「作ったのに気づかれない機能」というUXの典型的な欠陥がそのまま外部公開されている状態でもある。localStorageのみの設計上、バックアップ手段が無いまま外部ユーザーがキャッシュ削除に遭えば、全データを無条件に失う。

**最大価値**: 「採用済み」バッジと決定サマリーカードの視覚的な強調(緑のボーダー・背景)により、案件一覧を流し見するだけで「どれが終わっていてどれが未決定か」を一瞬で判断できる。

**改善案**:
1. `exportData`/`importData`をボタン化し、案件一覧画面にJSON書き出し/読み込み導線を追加する。
2. 決定済みの案件では「回答を追加」フォームを折りたたみ表示にする(SF-2)。
3. sourceプリセットに「Claude Code」を追加する(SF-1)。

## 3. 個人開発者

**良い点**: バックエンドなし・localStorageのみ・認証なしという技術選択は、個人開発者が7日で1人で完結させるスコープとして正しい。テスト78件・TypeScript・Vercel静的デプロイという構成もメンテコストが極小に保たれている。

**悪い点**: `exportData`/`importData`という「実装したのに繋いでいない」コードが本番に残っているのは、個人開発によくある「後で繋ごう」を忘れたまま公開してしまうパターンそのもの。関数はすでに存在しテスト済みで、実装コストはほぼゼロであるにもかかわらず、Day5・Day7と2回のdogfoodingを経てなお未着手なのは優先順位づけの見落としと言える。

**最大リスク**: 継続開発のモチベーションが「自分が使い続けるかどうか」だけに依存しており、外部フィードバックがゼロの状態でv0.2の機能追加に着手すると、自分好みの方向に閉じていくリスクがある(個人開発にありがちな「作者にしか刺さらない」ループ)。

**最大価値**: コードベースが小さく(画面2つ、CRUD一式のみ)、思いついた改善(source追加、export配線)を数分〜数時間で実装・検証できる状態を維持できている。ここまでの品質(lint・test・build全通過)を保っていれば、v0.2着手コストは低い。

**改善案**: 次にコードを触るときは、新機能より先に「実装・テスト済みだが未接続の機能(export/import)」を最優先で配線する。既存の作りかけを放置したまま新しいものを作らない。

## 4. SaaS創業者

**良い点**: 課金導線を意図的に後回しにし、無料での利用実績(継続利用意向)を先に見るという判断は、SaaS立ち上げの順序として王道。6件のdogfoodingを通じて継続利用意向は一貫して高水準にある。

**悪い点**: 「1つの意思決定につき1回しか使わない」道具である以上、利用頻度そのものが本質的に低い可能性が高い。実際、今回のdogfoodingでも3件処理するのに要した合計時間は数分程度であり、日常的に何度も開き直すタイプのツールではない。

**最大リスク**: DAU/WAUのようなエンゲージメント指標では成立しにくいプロダクトである可能性が高く、10人への案内後に「最初の数件は作るが、その後案件が増えずに離脱する」という形で失速するリスクが最も高い。今回の6案件もすべて既存の実務決定を後追いで記録したものであり、「新しい意思決定が発生した瞬間に自発的にCompare Noteを開く」という行動が本当に定着するかは、この検証だけでは分からない。

**最大価値**: 低頻度の利用であっても、「意思決定を記録する」という行為自体は特定の職種(コンサル・個人開発者・複数AIを使い分ける層)にとって代替品が乏しく、Day5・Day7とも継続利用意向が高い(体感値として毎回「したい」)という結果を引き出せている。

**改善案**: 課金はまだ時期尚早。利用頻度そのものを成功指標にするのではなく、「決定済み案件への再訪率」「再訪時に迷わず理解できたか」といった、低頻度利用を前提にした指標設計に切り替えるべき。

## 5. 投資家

**良い点**: 極小の開発コスト(7日・1人・バックエンドなし)で、実案件6件・継続利用意向が一貫して高いという定量データをすでに持った状態にあるのは、シード前のプロダクトとしては資本効率が良い。

**悪い点**: TAM(対象ユーザー)が「複数のAIを使い分けて、かつ意思決定の理由を残したい人」という非常に狭いセグメントに見える。しかも検証者が創業者自身(N=1)のみで、他者による需要の裏付けが一件も無いまま2回目の検証を終えている。

**最大リスク**: この製品は「複数AIチャットの回答比較+理由記録」という具体的な行動を前提にしているが、ChatGPT・Claude自体が今後マルチターンで「複数案を並べて比較する」機能をネイティブに持てば、このツールの存在意義自体が上流のAIベンダー側に飲み込まれるリスクがある。ローカルlocalStorageのみでバックエンドを持たない技術選択も、将来チーム共有や外部連携で収益化しようとした際の拡張コストを増大させる。

**最大価値**: 開発費・運用費がほぼゼロ(静的サイトのみ)のまま、既に6件の実利用データを持っていること自体が、追加投資なしで検証を継続できる資本効率の高さを示している。

**改善案**: 資金調達を検討する前に、最低でも自分以外の5〜10人分の実データ(利用頻度・継続率)を集めるまでは、この製品を「投資対象」ではなく「検証中の副産物」として扱うべき。ChatGPT/Claude自体の複数回答比較機能の進化を定点観測し、差別化ポイント(理由の記録・案件横断の一覧性)がベンダー機能に飲み込まれていないかを継続的に確認する。

## 6. 判定: Improve

**理由**:

- Phase1(6件の実案件dogfooding、Day5の3件+Day7の3件)でMust Fix該当のバグ・データ損失は0件。継続利用意向も一貫して高水準。コア価値仮説(比較→採用理由の強制記録→再訪時に迷わず思い出せる)は再現性を持って支持されている。**Pivot・Stopに該当する根拠は無い。**
- 一方で、(1) sourceプリセットにClaude Codeが無い問題(SF-1)がDay5(50%)→Day7(100%)で悪化しており、実利用で最も多いソースへの配慮が置き去りになっている。(2) `exportData`/`importData`という実装・テスト済みの機能がUIに一切繋がっておらず、README自体が「エクスポート機能なし」という(現状は結果的に正しいが本質的にはズレた)説明をしたまま外部公開されている。localStorageのみという設計上、これは外部ユーザーがデータを失っても復旧手段が無いという実害に直結するリスクである。
- いずれも実装コストは数十分〜数時間で、Pivotが必要となるような根本的な設計ミスではない。したがって「このまま10人に案内する(Continue)」のではなく、**上記2点を先に修正してから外部案内に進む(Improve)** のが妥当と判定する。

## 7. 改善案(Must / Should / Could / Won't)

### Must(3件以内、外部案内の前に対応する)

1. **`CANDIDATE_SOURCE_OPTIONS`(`src/types.ts`)に「Claude Code」を追加する**。Day5(50%)→Day7(100%)で悪化した実利用上の摩擦。実装は配列に1要素追加するのみ(5分未満)。
2. **`exportData`/`importData`をUIに接続する**。案件一覧画面にJSONエクスポート/インポートボタンを追加し、README.mdの「エクスポート・バックアップ機能は現時点ではありません」という記述を実態に合わせて更新する。関数・テストは既存のため、UIとファイル入出力(`Blob`/`<input type="file">`)の配線のみで完結する。

### Should(時間があれば)

- 「回答を追加」フォームの位置見直し(SF-2): 決定済みの案件では折りたたみ表示にする、または比較セクションの下に移動する。
- 案件一覧画面に「データはこのブラウザにのみ保存されます」という一言と、Must#2のエクスポート導線への案内を並べて表示する。

### Could(余力があれば、仕様の範囲内で)

- sourceプリセットに他の主要ツール(例: Cursor)を追加できる余地を残す。
- 比較候補の並べ替え(ドラッグ)。

### Won't(意図的にやらない、[issue-001_mvp_spec.md](./issue-001_mvp_spec.md) 6節の禁止事項どおり)

- クラウド同期・ログイン連携・チーム共有・通知
- 課金導線
- AIによる自動比較・自動要約・自動推薦
- 検索・タグ・フォルダ(案件数が少ない想定のターゲットユーザーには不要と判断済み)
