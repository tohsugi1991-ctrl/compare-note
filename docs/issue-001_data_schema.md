# ISSUE-001 データスキーマ

前提: バックエンドなし・APIサーバーなし・認証なし・データベースなし。すべてブラウザのlocalStorageに保存する。

---

## 保存方式

1つのlocalStorageキーに、3つの配列をまとめたJSONオブジェクトとして保存する(キーごとに分散させると整合性が崩れやすく、7日で作るMVPには複雑すぎるため)。

```
localStorage key: "compare-note:v1"

{
  "projects": Project[],
  "candidates": CandidateResponse[],
  "decisions": Decision[]
}
```

`v1`という接頭辞を付けているのは、将来スキーマを変える場合に無言でデータを壊さないため(マイグレーションはv1の間は不要。互換性のないスキーマ変更をする場合はキー名を`v2`に変え、旧データの扱いはその時点で個別判断する)。

---

## Project

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| id | string | ○ | `crypto.randomUUID()`で生成 |
| title | string | ○ | 案件名。空文字では作成できないようにする |
| context | string | - | 案件の背景を1行程度で。任意。JTBD_reportが指摘する「案件の文脈」を最小限だけ残すためのフィールドだが、詳細な資料化はスコープ外なので1行に収める前提でUIを作る(複数行入力は許可するが必須ではない) |
| createdAt | string(ISO8601) | ○ | 作成時に自動設定、以後不変 |
| updatedAt | string(ISO8601) | ○ | title/context編集時、候補追加時、決定保存時に更新 |

削除を検討したが残したフィールドはない。task側で提示された5項目はすべて一覧表示・更新日ソートに使うため必要。

---

## CandidateResponse(候補回答)

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| id | string | ○ | `crypto.randomUUID()` |
| projectId | string | ○ | 所属するProjectのid |
| source | string | ○ | どのAIか。UIはセレクト(ChatGPT/Claude/Gemini/その他)+「その他」選択時は自由入力。値としては自由文字列で保存する(将来AIサービスが増えてもコード変更なしで対応できるように、enumで固定しない) |
| content | string | ○ | 貼り付けたAIの回答全文。文字数上限は設けない(textareaのまま保存。極端に長い場合の見た目はCSSでスクロール領域にする) |
| order | number | ○ | 追加した順の連番(0始まり)。表示順の安定のためだけに使い、ドラッグ並べ替えUIは実装しない |

`order`は「必要性が低ければ削除」の対象として検討したが、削除するとJavaScriptオブジェクトの配列順(挿入順)に暗黙的に依存することになり、将来配列操作(フィルタ・ソート処理)を挟んだときに表示順が崩れる。1つのnumberフィールドを持つコストは低いため残す。

---

## Decision(決定)

1つのProjectにつき、Decisionは常に最大1件(上書き型)。決定履歴は持たない。

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| projectId | string | ○ | 対応するProjectのid。これがキーになる(1 Project = 0 or 1 Decision) |
| selectedResponseId | string | ○ | 採用したCandidateResponseのid |
| decisionSummary | string | - | 「採用した内容」の一言メモ。任意項目とした理由: selectedResponseIdとcontentだけでも「何を採用したか」は再現できるため必須にはしないが、複数の回答を組み合わせた・一部修正して採用した場合に補足できるよう欄は残す |
| reason | string | ○ | 採用理由。1〜3行を想定(ハード制限はかけず、UIのプレースホルダーとテキストエリアの高さで「短く書く」ことを促す程度に留める)。これが主要仮説(H1/H2)の検証対象そのものなので必須項目とする |
| decidedAt | string(ISO8601) | ○ | 決定を保存した日時。編集で上書きした場合は再更新する(「最後に決めた日時」を優先し、初回決定日時の保持はしない。履歴機能を持たない設計と一貫させるため) |

`id`フィールドは持たない。projectIdで一意に引けるため不要(タスク提示のスキーマ通り)。

---

## localStorageを使う場合の制約(明記が必須の項目)

- **端末・ブラウザをまたがない**: 同じ人でもPCとスマホ、ChromeとSafariでは別々のデータになる。同期機能は禁止機能に含まれるため、v1ではこの制約を仕様として受け入れる。案件詳細画面の目立たない位置に「データはこのブラウザだけに保存されます」という一言を表示し、ユーザーに誤解させない
- **ブラウザのキャッシュ削除・シークレットモードで消える**: 「サイトデータを消去」やシークレットウィンドウでの利用は、閉じた時点でデータが失われる。エクスポート機能は今回のスコープに含めないため、この制約も受け入れる
- **容量上限**: localStorageは概ね5MB程度(ブラウザ依存)。貼り付けるAI回答のテキスト量を考えると、通常利用で数百件の案件を作らない限り上限には達しない見込みだが、保存に失敗した場合(`QuotaExceededError`)は「保存できませんでした。データ量が多すぎる可能性があります」という最低限のエラー表示だけ用意する
- **バックアップ手段がない**: v1にはJSONエクスポート機能を含めない(禁止機能ではないが、5ステップの主要フローに不要なため今回は削る)。データ消失リスクはKill Criteria・フィードバック収集時にユーザーへ正直に伝える
